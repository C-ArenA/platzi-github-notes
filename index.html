Flujo de trabajo profesional: Haciendo merge de ramas de desarrollo a master

Buena práctica de flujo de trabajo

  1. Crear Ramas
  2. Asignar una rama a cada programador
  3. El programador baja el repositorio con git pull origin master
  4. El programador cambia de rama
  5. El programador trabaja en esa rama y hace commits
  6. El programador sube su trabajo con git push origin <nombre de rama>
  7. El encargado de organizar el proyecto baja, revisa y unifica todos los cambios

Para usar github con colaboradores, ellos deben tener una cuenta de github

Flujo de trabajo profesional con Pull Requests

El programador trabaja en su branch y puede subir a github en su branch, pero si quiere que sus cambios se reflejen en una rama más importante (podría ser master, pero por lo general hay un branch de staging previo al master) deben hacer el request (pull request) para que les acepten los cambios

El pull request es una característica de github no de git

Los que se encargan de esto son los devops
